1.1 Background Introduction

Inventory management, a critical element of store operations, is essentially the practice of ensuring store businesses have the right articles, in the right quantity, at the right time. This includes knowing when the store will receive and return articles from/to DC and/or suppliers, how much stock the store has on hand, how to decide when and what to order, and accurately track articles, all the while bearing store operation process.

Good inventory management will not only support the store, but also help headquarters successfully manage and monitor the inventory and improve store performance.

For store activities on a daily basis, staff need to perform goods receiving, return, gap check and perhaps stock adjustments if variance on articles available SOH is found.

More importantly, all store activities are recorded and can be generated in reports for supervisors or headquarters to review. Well-organized reports show the store's performance, which can help headquarters with further group decisions.

3.1 Inventory design

1) Design Background

DMALL system will pre-define 8 inventory locations for stores. And each store operation function/movement action will operate in a specific inventory location.

Each inventory location has its own inventory data. It will not be mixed up together so that we can clearly identify the current inventory situation and make proper decisions.

Each inventory location will be mapped with SAP system location.

2) Key design

Inventory location definition

============================================================
[1. Blueprint Document Scope > 3. Function Design Solution]

Inventory location code | Inventory location name | Definition
SLOC 0001 | Selling Area | Selling area inventory indicates the sales floor articles quantity. when POS selling, inventory should be deducted from the selling area.
SLOC 0002 | Offsite Warehouse | This location indicates the storage articles quantity in the outside warehouse
SLOC 0003 | Bad Order Location | Bad order location is only used for return scenario. these inventory should not be sellable or used for any business.store should separate into an individual area for display these articles.
SLOC 0007 | STDR from DC&Store | STDR location is only used for delivery discrepancy report scenario. these stocks should not be sellable or used for any business. Stores should separate an individual area for display these articles.

SLOC 0030 | Advance Order | Storage location for reserved items for advance orders
SLOC 0031 | Backroom | Backroom inventory indicates the storage articles quantity. when articles are replenished into sales floor shelves, users should perform the stock transfer function to adjust the inventory.

3.2 Good receiving from vendor (DSA)

1.AS-IS Overview

In SM current practice, RDU users will receive the DN, download and print out the PO document from SAP and perform itemized receiving. RDU users will perform the first check and note down the receiving quantity on the PO document. Counter checker will perform the second check and input the receiving quantity into FRS system. If there is differ between the first check and second check, manager will be involved to do the double check.

2.TO-BE Overview

============================================================
[1. Blueprint Document Scope > 3. Function Design Solution]

Goods Receiving from Vendor is the physical inbound movement of goods from an external vendor, either without PO or with PO, which results in an increase of stock on hand(SOH). In this section, it focuses on receiving with PO, including outright goods receiving, consignor with inventory goods receiving and receiving on non trade PO's for goods, supplies, collaterals, signages and services. Consignor without inventory good receiving please refer to stock adjustment function to increase the SOH.

One DSA PO, can be multiple received in the DMALL system. For articles from a vendor to stores, the receiving store performs a detailed physical count, checks and accepts the article based on PO for goods receiving, which article could be rejected if the article is not in the receiving PO or in poor conditions.

Over-receiving is NOT allowed based on business practice and system design. DMALL System only allows receiving goods less than or equal to PO quantity.

Free goods, which are performed GR as store stock but with zero value, will be shown on the PO as an individual article line. The receiving process of free goods is the same as normal articles.

1）End to End

In phase 1, PO creation happens in SAP system. DMALL will download previous day’s PO data at 5AM every day from SAP and save it in database. The reference order in DMALL is just a reference for SAP to generate the real good receipt. The reference order will be created when stocks arrive at the store.

DMALL only supports receiving goods existed in the Purchase order. Over-receiving and extra articles that do not exist in PO are NOT allowed in DMALL design. Stores need to handle the above situations offline.

For free gifts, it will be reflected in the PO. The store can receive it like normal articles. For free gifts which are not in PO, please refer to current business practice(SOP) to handle them. DMALL system only support receiving articles existing in the PO(in phase 1).

Purchase price is strictly controlled by merchandising department. Stores are NOT allowed to amend purchase price when receiving and DMALL also do NOT support it.

In phase 1, DMALL system only manages the inventory, and cost is managed by SAP system. DMALL system will NOT show any relative cost information.

For DSA order, if it is consignor with inventory PO, allow store team to receive it within 14 days after delivery date(PO ETA). If others POs, from DMALL system perspective, users can only receive before or on the expected arrivals date(ETA date). After the ETA date, stores are NOT allowed to receive. Actual receiving is only allowed by default on ETA, which will be controlled by SOP

3.5 Transfer to vendor/warehouse

Store raises a complicated RTV process and return to vendor/warehouse. based on pre-align logic to charge to vendor and dispose the unclaimed RTV order

2.TO-BE Overview

Goods transfer to Vendor/DC is the physical outbound movement of goods from store to Vendor/DC which results in a decrease in available SOH. It can be applied to below scenarios:

Each article item line should choose the corresponding return reason, DMALL system can support to maintain different reason code for clarifying the business purpose.

For returnable articles, the store could return the article back to Vendor/DC, and to be approved by the approval officer with a certain threshold quantity.

Once the goods are issued from the return store, stock will be deducted from the store and the Store available SOH will be updated.

For non-returnable articles and consignors without inventory articles, users should perform stock adjustment function to adjust inventory

System background introduction

To provide key information for customers to decide, article information, promotion information and price information will be displayed by price tag, according to different scenarios, it can be put on the shelf or directly tag on the articles.

TPS system provides Shelf tag to manage price tag information, simplify the price tag print and paster operation.

1. AS-IS Overview

Currently, the SM TPS system supports printing shelf label price tag and other tags like location rack, pallet ID, employment ID. Each price tag template is programed in the system in advance.

There are regular PCA, promotion PCA and Majorcon price label types, different label templates will be auto mapped by outright and majorcon articles.

Billing from the vendor by quantity when printing the price labels for consignment vendors.

2. TO-BE Overview

DMALL APP can be provided for store users to submit price label requests and below price labels will be supported to be managed in TPS system. Below is the TPS scope to be supported in DMALL.

Set paper size(mm), width and height

Enabling Status to control whether apply the paper or not

Quick copy of current paper format to edit and save to be a new paper.
