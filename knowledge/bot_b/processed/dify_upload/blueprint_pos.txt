[Function Design Solution > Installation & Start > POS Installation/Upgrade]

1. AS-IS Overview

Today legacy POS can be installed with an installation package and upgraded remotely with the store patch program.

2. TO-BE Overview

With the DMALL POS, the new POS machine with at least Windows 10 version needs to install the DMALL POS program manually with an installation package for the first time.

For further iteration, upgrades can be downloaded and installed remotely & automatically on each POS terminal individually via backend version release management.

3. TO-BE Overview

4. TO-BE Description

Page Screenshot | Function Description
Automatic Install New Version | Installation DMALL will provide an installation package to SM team, and we can manually install the DMALL POS on the Window 10 (and above) system by following the step-by-step guide. No manual input required. Start POS Application POS will be automatically started along with the Window System start. Upgrade Before a new version is released, DMALL team will align with SM IT team to fully test the version in the lab with testing environment. When a new version of the POS system is available and pass the SM team SIT/UAT process, the update process is as follows: DMALL implementation team notifies the SM team about the new version and its release note, and discuss the upgrade plan, including: Upgrade Timing Release method: general release or grayscale release DMALL will remotely push the new version to the POS system. The POS terminal detects the new release on the cloud and automatically initiates the download of the new version in the background without impacting sales. Update installation: Once download is finished, the POS system will be updated automatically during the next reboot.

5. Exceptional Scenario

[Function Design Solution > Installation & Start > POS Installation/Upgrade]

Typically, the POS program will initiate automatically upon the startup of the Windows system. However, if the POS is exited or fails to start automatically, store team members can locate the DMALL POS application icon on the desktop and initiate the program by double-clicking on it.

If a new version of the POS has been released online, the POS terminal will automatically download it in the background, ensuring uninterrupted sales operations. If the download is unsuccessful due to a network issue, the POS will automatically resume the download once the network connection is restored.

[Function Design Solution > Installation & Start > Roving POS]

1. AS-IS Overview

Today, all roving POS are kept in the HQ and the IT team needs to manually move the POS, POS Server to the store, waiting for BIR approval and then ask the vendor to finish the configuration.

2. TO-BE Overview

Moving to DMALL, with the device registration feature, any POS device can be effortlessly assigned into a certain store by changing the Store ID from the registration page mentioned above and eliminating the requirement for an additional POS server.

IT team just need to move the POS device to the store (no need POS server), waiting for the BIR approval and then change the store ID of each roving POS from the Parter System & reboot POS. No need to reload the master data & promo data as POS should read the article & promo info from the cloud.

Since the roving POS are not frequently used and when needed it might remain lower version, it is recommended to turn on the POS beforehand with a few hours with network connection, so that POS can start downloading the latest version and install the upgrade in the next reboot.

Upon successful store ID change, sales transactions conducted on this POS will be automatically synchronized with the store's POS server, eliminating the need for manual merging of data. (Pending BIR Confirm)

3. TO-BE Flow

4. TO-BE Description

SM needs to apply moving the POS from HQ to certain store with BIR and waiting for approval.

In the meantime, SM can move the POS from HQ to the store physically. No POS server is needed.

Once it gets approval, SM just needs to find the roving POS from the device list in the Partner System (see 3.1.2 above), edit the POS info and change the store ID (and Terminal ID if the current terminal ID is not unique in the target store) from the HQ to the target store. Click Save.

Store members can then reboot the roving POS, and the store ID on the POS top status bar should show target store rather than HQ.

============================================================
[Function Design Solution > Installation & Start > Roving POS]

All sales of roving POS will be counted on the target store rather than HQ. (Follow BIR Requirement)

For multiple Roving POS, each POS needs to change the store ID individually.

Since Roving POS can only be used for 15 days max, after 15 days the roving POS store will be automatically reset to HQ.

BIR Relate – Below are the expected design but might adjusted as per BIR advice.

The store address and TIN number on POS receipt of roving POS will also shows the target store rather than HQ (Follow BIR Requirement)

Roving POS Terminal Z will also be count in the target store (Follow BIR Requirement)

Roving POS grant total sale won’t be impact, it should be the total of all sales among various stores.

Since the BIR Combine report is per POS but not per store, the report should remain the same unless it meets additional requirements. (Follow BIR Requirement)

5. Exceptional Scenario

If reboot the Roving POS and found the store ID remains the HQ, please check the network connection of the roving POS as the store change require network connection. Once the network is restored, rebooting roving POS can successfully fetch the updated store ID.

[Function Design Solution > User Management > Prerequisite - Cashier Auto-deactive]

1. AS-IS Overview

The legacy system today two significant concerns: the inability to delete cashier accounts from the system after their leaving, as well as the lack of effective tracking to determine whether the accounts of departed cashiers have been deactivated or invalidated.

2. TO-BE Overview

SM team has expressed a requirement to pre-configure the expiry date of user accounts based on different roles during user creation. For example, a cashier role may have a validity period of five months, after which the user will be automatically deactivated, and manual activation will be required to continue using the account.

Users in a deactivated state will not be able to log in to any platforms (including Partner System, POS, OS App, etc.), but their data will still be retained in the system.

3. TO-BE Flow

Not Applicable

4. TO-BE Description

Clearly define the roles required by the merchants and add an expiry date to the role configuration. Users can choose not to set an expiry date or set it on a monthly basis.

All newly created users under this role will be affected by this expiry date. After the expiry date, the user's status will automatically change to deactivated and manual activation will be required in the Partner System to continue logging in to the Partner System or POS.

This configuration is only used to deactivate users and will not delete the user. The action of deleting a user still needs to be performed manually in the Partner System.

5. Exceptional Scenario

When a user is already in a deactivated state and is manually activated, POS needs to be in the network connected state to synchronize the activation operation.

[Function Design Solution > User Management > Cashier Login/logout]

1. AS-IS Overview

Today the cashier is able to login via either manually inputting the account and password or scan a barcode on the employee card to quickly login. Also, the cashier is not allowed to log out until all suspended orders have been addressed.

2. TO-BE Overview

DMALL POS provides two login methods: account and password login or scanning the QR code of the DMALL OS App for login. Currently, SM will exclusively utilize the account and password login method, as the store team members are prohibited from using mobile phones. But the QR login function is there in case we need it in the future.

For logout, DMALL POS follows the as-is logic, the cashiers could log out when there are no ongoing sales transactions and no suspended orders.

3. TO-BE Flow

4. TO-BE Description

Page Screenshot | Function Description
POS Login Page | Login POS Upon launching the POS system, input the appropriate cashier account and POS login password into the designated fields on the login interface. Subsequently, click on the <login> button to proceed. While login, if there are suspended orders in the POS right now, then cashier is not allowed to login until he/she gets the supervisor authorization on POS. Log Out POS Once the cashier has successfully logged into the POS, he/she can access the system menu by clicking on the DMALL POS logo located in the upper left corner. Within the system menu, the cashier can initiate the logout process by selecting the "Logout" option. While logout, if there are unfinished suspended orders, cashier is not allowed to logout until he/she clear out all these suspended orders. Exit POS Program Once the cashier has logout and goes back to the login page, if he/she wants to exit the POS program, he/she proceeds to click on the <Exit> button located in the upper right corner to exit the POS program.

5. Exceptional Scenario

Cashiers should be able to login & logout if the POS has no network work connection as there is an offline database.

Input the user account and password of the authorizing personnel.

Scan the QR code of the authorizing personnel from the DMALL OS App (Please note that SM currently does not support this feature).

[Function Design Solution > Sales > Pre-order]

1. AS-IS Overview

Today there are two types of pre-orders for the SM:

Normal pre-order for the customer: the customer will pay for a deposit article for a certain amount (determined by the cashier). Then for the 2nd transaction, the cashier needs to present the deposit sale receipt so that the deposit amount will be adjusted accordingly. In this way, the deposit amount has to be used fully in the 2nd transaction, and no change for the exceed deposit amount in the 2nd transaction.

Pre-order for corporation: Sometimes the corporate prefers to deposit a certain amount with the Treasury office and then use a deposit for multiple transactions.

Note: Deposit should just be a part of the whole pre-order business. After the deposit, the store needs to prepare the article ordering, inventory lock later.

2. TO-BE Overview

Pre Order is a type of transaction currently offered by SM Market. Customers have the option to prepay a deposit for items that are not yet in stock. Upon arrival of the items, customers can use their deposit ticket to prioritize their purchase eligibility and offset the payment with the prepaid deposit.

Usually there are three approaches for the pre-order scenario:

(SM AS-IS) The POS system adds a negative service item to deduct the deposit amount.
Not recommended as negative service items are not commonly used, especially from a financial perspective. Additionally, it does not accurately reflect the pre-order business on the system level, as shopping carts should only include articles or services that are purchased by the customer.

============================================================
[Function Design Solution > Sales > Pre-order]

Finance should create a new type of order called "Deposit Order". Deposit orders should not be considered as sales since the income is not technically realized until the customer pays the remaining amount. When a customer uses their deposit to settle the second order, the deposit order will be automatically deducted to maintain balance between the second order and the deposit payment.
This approach should be the best practice, but it needs a lot of modification from the finance side and impacts the reconciling process. It's recommended to apply this solution once SM adopt the DMALL ERP system and uses the DMALL finance module.

Based on SM's current situation and best practices, it is recommended to simplify the second option for pre-order transactions. Pre-paid orders will be processed as special sales orders (not post to BIR) with an additional indication as pre-paid orders. When customers use their deposit to pay for the second order, the deposit amount will be deducted as part of the payment, and the payment will be associated with the pre-paid order.
The advantage of this approach is that it provides a clear and intuitive process for both cashiers and customers. Furthermore, this approach does not require any modifications from the finance department, making it more feasible.

For the two types of the Pre-order in SM, we recommend following solutions:

Normal pre-order for the customer:

The first transaction to collect deposit will be recorded as a service transaction. The deposit item has to be sold solely without mixing with other regular items. And the deposit amount is manually inputted by the cashier. Eventually, the customer will get a deposit receipt with a QR code on it.

2. TO-BE Overview

With DMALL POS, after a customer completes SMAC login and successfully pays for an order, the POS system will use the point issuance API provided by DAC to issue points to the SMAC member in (almost) real-time.

3. TO-BE Flow

4. TO-BE Description

Cashier needs to complete the SMAC member login during the sales process (membership login is also available when POS has no network connection).

The POS system calculates the eligible points based on DAC's rules.

After completing the payment for the order:

If the POS system has internet connection, it will immediately initiate a request to issue points.

If the API requests success, the points will be issued to the SMAC member directly.  Customers should expect to receive the points within 15 minutes after completing the order.

However, if the API request fails or times out, there will be no prompt on the POS side (as the order has already been completed on the POS).

If the POS system does not have internet connection after payment, it will temporarily keep the point request locally and call the API when the POS is reconnected to the internet. 
In this case, customers will receive the points after a certain period after completing the order (depending on the duration of POS disconnection).

There will also be a daily feedback file sent to DAC, which will include all the orders. It is recommended that DAC handle any failed or timed-out point requests as a fallback solution (confirmed with DAC on 7th Dec 2023).

5. Exceptional Scenario

Reconciliation failed

If reconciliation fails for a certain store in a certain day, the EOD data posting record will fail and can be viewed in backend system. Normally, the reason will be the network exception, which some of the transactions haven't been able to synchronize to the cloud server. IT should manually export the transaction data from in-store POS server and import it to cloud server. After that, IT can trigger EOD in the cloud server manually without reconciliation.

POS does not connect to in-store POS server

There is scenario that POS does not connect with in-store Server during the daily sales (move to the mall outside the market). For the SOP this POS has to be move back to the market and connected to the in-store POS server before closing the daily sales or EOD trigger time.

If the employee forgets to move the POS back to the store, There is a chance that the reconciliation passed when this POS does not successfully send any transaction to cloud server within this day (otherwise the reconciliation will not pass because there are more transaction in cloud compared to in-store POS server), therefore the EOD data posting is complete without the transaction in this POS.

In this case: 1) CS personnel and IT manually copy the transaction files from POS and paste to cloud server; 2) SAP reverse the whole store posting for this day; 3) IT manually triggers the EOD data posting again from backend.

POS physically broken

Firstly, make sure the hard disk still works, at least the transaction files can be copied from POS.

Secondly, manually paste the files into cloud server and manually trigger the EOD data posting.

[Function Design Solution > Other Cases > Offline Service]

1. AS-IS Overview

SAP and other related HQ systems synchronize master data such as articles, promotion, etc. through files to store telnet server and then to POS server. Then the POS server synchronizes data to the POS terminal to support POS business flow.

Daily transaction data from POS terminal will be synchronized to POS server and generate files to send back to SAP and other related HQ systems through telnet server.

2. To-BE Overview

As DMALL system is cloud based, POS terminal can directly request cloud to use latest data to process transactions while the transaction data can be synchronized to cloud server in real-time as long as the network is normal. However, due to the complexity of the store network situation, an offline service strategy is needed to ensure normal POS sales when the network is disconnected.

Offline master data synchronization

Any master data change will generate a new offline data version and synchronize to POS offline service data. Both delta and full-data sync are supported.

For BCP strategy, offline data can be exported from cloud server and imported to the in-store POS server. POS offline service can fetch offline data from in-store POS server.

Transaction data posting

POS transaction data will be synchronized to cloud server in real-time as long as the network is connected. If the network is disconnected, transaction data will remain in POS upload task queue and synchronize immediately until the network is restored. The transaction data will also be synchronized to the in-store POS server for back up.

For BCP strategy, if the store network is disconnected for a long time, transaction data can be exported from in-store POS server and imported to the cloud server.
