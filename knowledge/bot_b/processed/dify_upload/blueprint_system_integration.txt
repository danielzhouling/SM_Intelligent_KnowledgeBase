1. Blueprint Document Scope

1.1 Background Introduction

This document is for SM digitalization transformation project. This project is divided into 4 phases, and this document is for Phase 1A, and in this phase, DMALL will help SM to switch POS and Inventory related systems. As defined in the project scope, the proposed solution includes feature support for POS, Inventory, Master data structure, Picking and Dashboard.

To support these features, system integrations are needed to provide data exchange between DMALL OS and multiple SM Legacy systems, this document will provide solutions for the system integrations, which will be mainly focusing on these aspects:

Master data, which is the basis for the project.

Inventory(Store Operations Interaction flow), which is for inventory related support features.

End of Day(EOD), for compatibilities with Legacy or External system interactions.

3.4 Technical Considerations for To-Be Integration Design

3.4.1 Data Integration Approach：

SFTP (Secured File Transfer Protocol)

For the compatibilities with SM existing systems and data flows, DMALL will follow SM existing FTP conventions as long as they are considered safe and secure.

SM provides the SFTP Server for data integration.

For the new FTP based data exchange, we suggest both SM and DMALL follow the following rules:

No. | Rules contents
1 | Both parties agree to the file uploading directory, such as /upload/business.
2 | The party who writes the data, first creates a temp directory in the above directory.
3 | Write all the data files in the temp directory or its subdirectories.
4 | After all the files are written, the operator needs to rename the temp directory to the final directory.
5 | Both parties agree on the data reading time (UTC+8). Data modification is allowed before the reading time. Data updates after the reading time need to be manually processed after the data reading party confirms the updates.
6 | Before the data is read, add the current processing timestamp as a prefix for all files in the final directory, and then move the files to the bak directory to prevent the data from being modified during the reading process, the files which timestamp is before 31 days in the bak directory will be archived to ./bak/${YYYYMM} directory early morning on the first day of every month, and the retention period is half a year. After the data is processed, if there is still content to be processed in the final directory, the content is considered as a change after the file reading and needs to be processed manually.
7 | If multiple files need to be uploaded, it is recommended to compress the files first before uploading. The .zip compression format is recommended.
8 | Unified UTF-8 encoding is required for files.
9 | Merchants who want to access the SFTP server need to apply for an account and whitelist from DMALL.

3.4.2 Performance &  Stability Consideration

Running performance & stability monitoring:

Each DMALL OS system will add event tracking and monitoring in key links such as interface interaction, database access and file transfer, collect relevant performance indicators in real time.

And will notify the IT maintainer of DMALL OS system through a unified alarm mechanism if the link exceeds the threshold, so as to intervene in troubleshooting problems and eliminate hidden dangers at the first time.

============================================================
[System Integration]

DMALL IT maintainers will operate on the issues as soon as possible when the situation occurs. And we will notify SM the first time to solve the problem together if necessary.

The interaction of interface data and transmission of file contents needs to be completed within the time limit agreed by both parties, subject to the detailed integration plan.

3.4.3 Security Consideration

SFTP:

As SM provides the SFTP server, we recommend that SM IT team ensure the security and high availability of SFTP server, including but not limited to creating individual username/password or SSL certificate for integration to authenticate.

Encrypted communication based on SSH protocol can ensure the communication content will not be eavesdropped on by the third party.

Where applicable, DMALL will follow SM IT standards utilizing Service Accounts.

Https: Use the SSL/TLS protocol for encryption and authentication. And Https Use port 443 by default instead of HTTP's port 80.

JSON based API adopts OAuth user authentication and authorization rules.

3.4.4 Exception Handling Consideration

Under the premise that all the infrastructure such as hardware, network and others work normally, the Exception Retry Mechanism will be applied for the following scenarios automatically:

API Interface failure

SFTP files transfer failure

JCo interface failure

During the data exchange processes, when Master Data are sent from SAP to DMALL OS and Inventory related API call from DMALL OS to SAP, the summaries should be logged. And for EOD files(any periodically sent files), when data are sent from DMALL to SAP the summaries should also be logged. During these data transmissions, errors might occur, any abnormalities, failures and exceptions should be caught, logged and sent in an email. The recipient should include the related email groups of business and IT team members.
